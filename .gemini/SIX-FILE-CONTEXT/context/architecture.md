# Architecture Context

## Stack

| Layer            | Technology                                | Role                                                                           |
| ---------------- | ----------------------------------------- | ------------------------------------------------------------------------------ |
| Framework (UI)   | Next.js (App Router) + TypeScript         | Frontend application, UI state (governance checklists, physical state machine) |
| UI Library       | Tailwind CSS + shadcn/ui                  | Styling, high-contrast dark mode, and accessible components                    |
| Backend          | Node.js + Express                         | API orchestration, Predictive Demand Engine (burn-rate math), security         |
| AI Engine        | Gemini 1.5 Pro (Vertex AI SDK)            | Copilot for mitigation strategies, surplus mapping, and pipeline drafting      |
| Data Integration | Fivetran REST API                         | Automated infrastructure provisioning for emergency inventory transfers        |

## System Boundaries

* `duronet-client/` — Owns the Next.js frontend, UI state, and API client. It strictly handles presentation, user interactions, risk visualizations, multi-stakeholder approval checklists, and the 4-stage physical transit tracker.
* `duronet-server/` — Owns the Node.js backend. It is responsible for executing the deterministic predictive math (`/api/inventory/predict`), securing API keys, running the Vertex AI/Gemini orchestration, validating Fivetran JSON schemas, and securely deploying live pipelines via the Fivetran API.
* `data/` (within server) — Owns the mock static JSON files representing real-time hospital telemetry, drug shortage forecasting, and regional supplier inventory breakdowns (Total vs. Committed vs. Available).

## Storage Model

* **Mock Database (JSON):** For the hackathon MVP, localized hospital inventory, patient influx telemetry, and regional hub surplus data will be stored as structured JSON.
* **Transient State & Artifacts:** Fivetran configurations generated by Gemini, multi-stakeholder notification arrays, and transit timelines are temporarily held in server memory or client React state until the logistics sequence is completed.
* **NO PHI Storage:** The system absolutely does not connect to or store any hospital Electronic Health Records (EHR) or individual patient identities. Telemetry is restricted strictly to aggregate numbers (e.g., "85 respiratory patients").

## Auth and Access Model

* **Mock Authentication:** Real authentication is out of scope to save time. The app will hardcode a simulated "Supply Chain Director" session.
* **Access Control & Governance:** The UI assumes the active user has administrative rights but enforces a simulated multi-stakeholder governance model. The execution of critical infrastructure requires verified coordination (pinging) of defined roles (e.g., Pharmacy Director) before the system unlocks action privileges.

## Invariants (Strict Rules)

1.  **Decoupled Isolation:** The Next.js client must never contain backend logic or secret API keys. The Node.js server must never render UI components.
2.  **Zero-PHI Boundary:** The application and AI models must never ingest, mock, or process patient data. The predictive model acts strictly on aggregate telemetry and SKU burn rates.
3.  **Human-in-the-Loop Enforced:** The system must *never* execute a Fivetran data pipeline autonomously. The AI's final output is always a *drafted* configuration payload that requires multi-stakeholder clearance and an explicit user click ("Approve & Deploy") to execute.
4.  **Data Event ≠ Physical Event:** The system must recognize that an API deployment does not equal physical delivery. Successful data pipelines initiate a visual 4-stage state machine (Pipeline Deployed → Transfer Authorized → In Transit → Physically Received). Map status markers cannot resolve to "Safe" until physical receipt is confirmed.
5.  **Latency Handling:** All client-side interactions involving the AI agent or Fivetran API must implement clear loading states (e.g., skeletons, stepper progress) to account for generation times and execution latency.
